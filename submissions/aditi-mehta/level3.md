### Agent Description: The SMILE Gap Analyzer
Instead of a basic chatbot that just answers the user question, I built the **SMILE Gap Analyzer (The Critic Agent)**. It acts as a Senior Architect. When a user submits a digital twin idea, this agent doesn't just answer but audits the concept against the SMILE framework to find physical, human, and architectural blind spots (like hardware failures or environmental risks).

### Choices Made Beyond the Instructions
The instructions implied keeping a persistent `stdio` stream open, but I noticed this frequently resulted in a `BrokenPipeError` due to process syncing issues. 
* **Stateless IPC Bridge:** I chose to go beyond the instructions by engineering a stateless bridge using Python's `subprocess.communicate()`. Every tool call spins up the Node process, retrieves the JSON-RPC data, and closes cleanly. It is slightly heavier on compute but makes it significantly more reliable.
* **Security Layer:** I didn't want to directly pass terminal arguments to an LLM. I implemented a Regex-based sanitization layer to clean input for special characters, mitigating basic prompt injection and terminal escapes before the MCP server is even touched.

### Evidence of Explainability & Provenance (XAI)
If a user asks my agent, *"Why did you recommend this?"*, they don't even have to ask—the agent is hardcoded to tell them upfront. To prevent the LLM from hallucinating, I implemented **Few-Shot Template Forcing**. The system prompt explicitly restricts the local model to the real LPI tool names and forces it to cite its data source for every single critique.

**1. Inline Citations:**
Every architectural critique generated by the agent is front-loaded with its exact source. Here is the exact output showing strict explainability:

```text
 MISSING REALITY REPORT 
==================================================
[SOURCE: LPI/smile_overview] Overview of the downtown traffic intersection suggests potential safety hazards such as pedestrians and bicycles, but neglects consideration for emergency vehicles needing priority access.

[SOURCE: LPI/query_knowledge] Knowledge base indicates staff manual overrides are not addressed in the concept; therefore, ensuring accessibility without compromising functionality is crucial. 

[SOURCE: LPI/get_case_studies] Case studies reveal power outages can affect sensor accuracy; thus, developing a robust backup power supply for sensors should be prioritized.
==================================================
```
